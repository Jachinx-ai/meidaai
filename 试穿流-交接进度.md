## 4. 核心架构变更：试穿引擎换成 gemini-3.1（已提交 faf7606）

**为什么换**：旧的 aitryon-plus（阿里百炼 VTON 模型）有两个硬伤——① 输入格式写死只支持"人+上装+下装"，**鞋子根本塞不进去**；② 而且它在旧代码里因为上传缺 MIME 类型，其实 100% 失败在回退叠图。

**做了什么对比测试**（结论，供你判断，不用重测）：
- gemini-3.1-flash-image（Nano Banana 2）：一步喂"模特+上衣+下装+鞋子"→ 生成整套上身图，**鞋子能生成、且保住模特本人的脸**。~¥0.5/张。✅ 选它。
- gemini-2.5-flash-image：更便宜(~¥0.28)，但**会把模特换成另一个人**（换脸），不能用。
- 阿里 shoemodel-v1（专门试鞋）：换鞋好，但整图重绘也会**轻微换脸**；且不在 OpenRouter。
- 全套换其它 VTON 模型：市面没有"一步连鞋一起生成"的成熟 API，放弃。

**怎么改的**：
- `server/ai/config.js`：`MODELS.tryon` 从 `"aitryon-plus"` 改成 `"google/gemini-3.1-flash-image"`。
- `server/ai/tryon.js`：整个重写。走 OpenRouter，把 模特图+上衣/连体裙+下装+鞋子 组成多图消息，一步生成。契约不变（输入 `{modelImage, items}` → 输出 `{image|null}`，null 时前端叠图兜底）。
- `tryon.html` 三处配套（已在 faf7606）：genImage 分支不再前端叠鞋（鞋子已在生成图里）、缓存 key 重新纳入鞋子、快照直接用生成图（`compressImage`）。

**连带影响 / 已知副作用**：
- 默认模特（`assets/real/model-f.png`）现在也能真生成了（旧模型要求真人照片）。
- gemini 是"理解后重画"：纯色/基础款还原好；**复杂印花/logo/文字可能会糊或变形**；每次结果有细微随机差异。这是生图模型的固有边界，不是 bug。
- 提示词在 `server/ai/tryon.js` 里内联（不在 prompts.js）。要调效果就改那段中文 prompt。

## 5. 关键机制 / 前人踩过的坑（改代码前先懂这些）

- **防烧钱三件套（勿删）**：换装后 600ms 防抖（`applyTryon`）；同模特+同一身衣服结果内存缓存（`genCache`）；过期结果按序号 `tryonSeq` 丢弃。加载态改动已把"命中缓存"做成秒显不重复计费。
- **加载态状态机**（改完后）：`renderStage()` 优先级 = 有生成图 `genImage` > 生成中 `generating`（加载态）> 兜底叠图。
- **鞋子**：模型已能生成鞋子，前端**不要再叠鞋**（faf7606 已删叠鞋逻辑）。
- **舞台底色**：`syncStageBg()` 自动取当前图片边缘色，别写死颜色。
- **localStorage 5MB 上限**：所有进 `Store` 的图必须先 `compressImage`。换 gemini 后每张快照都是生成大图，**超限风险更高了**（这是待修问题，见下）。
- **后端"假离线"**：前端探测后端只等 1.5s，超时静默降级本地叠图。"AI 没生效"先确认 `npm start` 还活着。

## 6. 待解决问题清单（带解决思路）

### 💬 待团队/产品决策（不写代码，等结论）
1. **bodymask 去留**：见第 3 节。换引擎+加载态后影响面已很小（只剩失败兜底）。选项：留着当失败兜底 / 也删掉让失败时显示纯模特图。
2. **防连点最终交互**：现在是"锁按钮"（临时，c76482b）。用户想要的是"点生成后跳转生图页、锁整页"。定了再改。
3. **创建模特两条路径不一致**：上传照片成功停原页，选预设却自动返回，要统一。
4. **上传提示的"错误示范"图**：`create-model.html` 里坏例子全是同一张剪影乱变形，语义不达意，需设计出真实示例图。

### 🔴 待修 · 高
1. **切模特后返回试穿页不刷新**：`tryon.html` 无 `pageshow` 处理 + `Store._cache` 内存缓存，手机 bfcache 返回时台上还是旧模特。思路：加 `pageshow`（`e.persisted`）监听 → 失效缓存 + `renderAll()`。页内可修，不碰公共文件。
2. **localStorage 超限无保护**：`Store.set`（在公共 `js/app.js`）直接 `JSON.stringify+setItem`，撑爆会抛异常中断流程。换 gemini 后快照更大、风险更高。**改公共文件要走刘倩**；页面层可先包 try/catch 兜底。
3. **create-model 自动返回无防重入**：`setTimeout(history.back)` 双击会连退两级、跳过试穿页；直开本页点"确认"无反应。思路：加进行中标志 + `history.length>1` 兜底。

### 🟡 待修 · 中
4. 生成失败的组合不缓存 → 同一失败组合反复重试**反复计费**（gemini 下依然成立）。
5. 删除当前模特照片后，页面显示的照片 ≠ 实际生效模特（`create-model.html` renderMine 显示 mine[0] 但 curModel 置 m1）。
6. `?outfit=` 旧链接绕过性别/删除过滤，还能产生"永远看不见的幽灵收藏"；无参进入固定穿女装第一套。
7. `wearing` 为空也发生成请求并闪"约需20秒"（应在无衣服时前端直接跳过）。
8. `?item=` 从衣橱进来绕过"须先上传"检查。
9. 图片压缩失败时原图直存（放大容量风险、坏图能建成模特）。
10. 删除模特照片无确认弹窗（删衣服有，全站不一致）。
11. 试衣间查看器切缩略图全量 `innerHTML` 重建 → 真机卡顿。思路：只换主图 src + 切 `.on` 类。
12. create-model 多照片管理缺口（只能看/删当前一张）。

### 🟢 待修 · 低（攒一波清）
名称未转义拼进 alt、`syncStageBg` 竞态、穿搭 tab 空态、本地兜底搭配不分性别（公共 ai.js）、全量重渲染性能、横屏适配、空列表可进编辑态、`FileReader` 无 `onerror`、老安卓 EXIF 方向。
